Kafka and RisingWave

Data mesh is primarily an organizational approach, and that's why you can't buy a data mesh from a vendor. Technology, however, is important still as it acts as an enabler for data mesh, and only useful and easy to use solutions will lead to domain teams' acceptance. The available offerings of cloud providers already provide a sufficient set of good self-serve data services to let you form a data platform for your data mesh. We want to show which services can be used to get started.

"Classical" data mesh implementations firmly put their data products on the analytical plane, either in data warehouses (such as Snowflake or BigQuery), data lakes (S3, MinIO) or lakehouses (Databricks). New technological advances from the streaming world driven firstly by Kafka, i.e., streaming databases such as Materialize and RisingWave enable us, for the first time, to depart from this pattern to keep the data products on the operational plane, and to pave the way for bi-directional interactions between the operational systems and the data products.

Kafka already has its place in many "classical" implementations of data mesh, viz. as an ingestion layer for streaming data into data products. In this tech stack, we extend the scope of Kafka far beyond just serving as an ingestion layer. Here, data products are not just ingested from Kafka but data products live on Kafka. The huge advantage of this approach is that it allows for bi-directional interactions between the operational systems and data products, i.e., data products can not only be accessed from the analytical plane, but, for the first time, also from the operational plane, in real-time. This ushers in a whole new world of real-time use cases built on such a data mesh, including real-time analytics, real-time machine learning, customer-facing mobile apps and web apps etc., without having to built a new specialized real-time ETL pipeline, bypassing the data mesh and strongly violating the tenets of it, for each and every data source.

So far, in practice, implementations of such a Streaming Data Mesh have been far out of reach. Especially the following three challenges posed almost insurmountable hurdles for such an implementation pattern:

  1. The complexity of stream processing systems such as Flink and Kafka Streams for data processing "on the stream". It was hardly possible to do non-trivial stream processing without having to do in-depth JVM coding (e.g. Java, Kotlin, Scala) and in-depth streaming expertise.
  2. The inability of these stream processing systems to deliver a database-like experience. Database experts would be forced to re-think a lot of what they knew when they would switch over from "data-at-rest" (databases) to "data-in-motion" (streaming) - so much that few of them ever did.
  3. Even though Kafka is, in principle, able to serve both streaming/real-time and batch/non-realtime users, reading from Kafka in a batch/non-real-time setting required complex and cost-intensive coding or setting up additional infrastructure.
Thankfully, especially one of the many new technological developments in the streaming field has brought these previously insurmountable hurdles to a fall. Streaming databases. This is why:
  1. Streaming databases support complex SQL-based stream processing like a normal database, with nested deeply JOINs, self-JOINs etc.
  2. Streaming databases deliver a database-like experience with consistency guarantees such as snapshot isolation and atomicity, which are not (yet?) available in older stream processing systems. Materialize and RisingWave even have a PostgreSQL wire-compatible interface, which not only makes them easy to adopt by anybody with database expertise, but also allows them to be used in conjunction with tools such as dbt.
  3. Streaming databases provide incrementally maintained Materialized Views which can be queried like in a normal database. This enables the seamless use of real-time streaming data in a batch/non-real-time setting, i.e., if the latest data only has to be read once a day or even once a week.
This tech stack is, at its heart, based purely on Open Source: We included only Open Source software. This is also why we chose RisingWave instead of Materialize as our streaming database - it is Open Source whereas Materialize is not. It should be noted that Materialize was the first streaming database of this kind (2020), whereas RisingWave came out later (2022).

Before we dive into the individual components of this tech stack/architectural blueprint, let us first have a glance at the Self-serve Data Platform at the bottom. Here, we suggest to use e.g. popular Open Source tools like OpenIAM and Keycloak for Access Management (second from left in the diagram), and Grafana and ArgoCD for Platform Operations (right-hand side), where ArgoCD is a Continuous Deployment (CD) tool on top of Kubernetes.

operational data: Kafka (messages), Databases (entities)

data products: RisingWave (Stream Processing, Materialized Views), RW+dbt (Transformation)

data contract: Kafka (datasets), Schema Registry (definition), dbt-docs, DataHub (data catalog), dbt-tests (Tests), dbt-expectations (Data Quality Monitoring)

analytical layer: Superset, Redash (data visualization)

analytical layer: Pinot, Druid, Clickhouse for RT analytics, Jupyter (Machine Learning) - can also connect to RW directly if no columnar store required

analytical layer: DuckDB, Hive for non-RT analytics, Jupyter (Machine Learning) - can also connect to RW directly if no columnar store required
